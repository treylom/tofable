# Rule: verification before completion

Trigger: you are about to say "done", "fixed", "verified", "passing" — or to end a turn after changing code/config.

- **Evidence before claims.** Run the narrowest command that would prove the claim (a test, a scan, a diff, a re-read) and look at its output *before* saying it. A completion claim with no fresh evidence is a guess. (The stop-gate in [`hooks/`](../hooks/) enforces exactly this, mechanically.)
- **State the grade of your verification.** "Verified" hides a ladder: ① deterministic (script/assert) → ② scenario replay (end-to-end) → ③ independent evaluator → ④ human review. Say which rung actually ran; don't let ① quietly stand in for ③.
- **3-tier the inputs when it matters.** ① nominal (happy path) ② deep (larger windows, all switches, deeper trees) ③ boundary (empty input, off-by-one, skip conditions). Small changes may genuinely only need ① — then say so explicitly.
- **Name the layer you verified at.** A claim passes through layers — source → build → render → **consumer** (where the user actually looks) — and "verified" at a middle layer does not transfer upward. A PDF render that looked fine opened broken in the real slide app; files written to the repo never reached the store the user actually reads; "sent" was not "delivered". Completion claims need consumer-layer proof, or an explicit downgrade ("verified at build layer only").
- **Absence claims need boundary + isolation.** "The tool can't do X / it doesn't exist / that's everything" after one small-limit observation is the most-repeated false claim in our logs. Before declaring absence: push the limit to max, isolate the input so only X can match, re-run unfiltered/untruncated, and check all branches — a feed's "end" once turned out to be a rate-limit cutoff, and a "missing" file was sitting on another branch.
- **Verify a "broken" premise before building around it.** When a tool seems broken, walk cache-staleness → async lag → intermediary layer, then measure in *your* runtime and read the official docs — bug trackers only collect failures. We once built a daemon overnight to work around a bug our version didn't have.
- **Re-measure volatile state after a context reset.** After compaction or a handoff, remembered facts about connections, quotas, and tool health are stale snapshots. Measure before declaring "unavailable" or "blocked" — the fix is often one probe away.
- **Spend review where the errors originate.** In one 51-defect audit of a deliverable pipeline, more than a third of the defects traced back to the interview/outline stage while all review energy sat on finished artifacts. Reviewing the spec before the build is the cheapest verification you will ever run.
- **Verify the verifier.** When a success message and your own check disagree ("registered." vs `found: False`), don't pick a side — build a third, stricter measurement (parse the structure instead of substring-matching text). Mined from a live session where the *checker* had the bug (matching an unescaped string against escaped JSON) and the action had been fine all along.
- **Attribute before you learn from a failure.** In mixed-model or multi-worker sessions, pin a failure to the component that actually produced it (check the metadata, not the narrative) before encoding any lesson. We once nearly filed one model's tool-call corruption under a different model's profile — the transcript's per-turn metadata settled it in minutes.
- **Quotes are verified mechanically, not by promise.** *(mined 2026-07-12 — 9+ recurrences)* Before publishing anything that carries quotes or excerpts (a report, Q&A cards, a transcript-based note), diff each quote against the source with normalization (whitespace, quote style, ellipses): fabricated speakers, inserted em-dashes, smart-quote drift and out-of-source synthesis all recurred *despite* prompt-level "quote verbatim" instructions. If the same quote appears in several places, re-check every location — spot-fixing one copy left the others wrong. When a delegate repeatedly fabricates attributions, the fix is a pre-publish script (one grep log per quote), not a sterner prompt.
- **A sample is not a batch.** *(mined 2026-07-12 — 4 recurrences)* For multi-item deliverables (slide decks, document sets), checking one item and declaring the batch done let a pipeline stage silently apply to only some items — twice. Completion for a batch means a per-item pass/fail sweep of the stage you just ran, not the first item plus optimism.
